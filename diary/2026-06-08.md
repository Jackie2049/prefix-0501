# 2026-06-08: 16-layer GRPO E2E Training Breakthrough + PG 竞品分析

## Summary

16-layer Qwen3-27B GRPO E2E training SUCCESS on 24 GiB RTX 4090 GPUs!

The key breakthrough: **CPU Adam step** — a manual AdamW optimizer that runs entirely on CPU, bypassing FusedAdam. Peak GPU memory = only bf16 params + activations (no optimizer state on GPU). This is what made 16-layer training possible.

## Problem

Megatron's `optimizer_offload` mechanism moves optimizer state (fp32 main_params + exp_avg + exp_avg_sq) from GPU to CPU AFTER creation. But during the optimizer step, ALL state must be on GPU simultaneously because FusedAdam requires it. For 16 layers (8.3B params, 2.08B per GPU with TP=4):

- fp32 main_params: ~8.32 GiB per GPU
- exp_avg + exp_avg_sq: ~8.32 GiB per GPU
- Total optimizer state: ~16.64 GiB per GPU
- Plus bf16 params (~4.16 GiB) + activations → >24 GiB → OOM!

Previous attempts:
1. Force FusedAdam init before offload → OOM (GPU already full)
2. CPU-allocated optimizer state → cudaErrorIllegalAddress (FusedAdam CUDA kernels can't access CPU tensors)
3. Deferred load (load optimizer state only before step, unload after) → Still OOM for larger models (8 layers barely fits, 16 layers impossible)

## Solution: CPU Adam Step

Implemented `cpu_adam_step()` in `verl/utils/megatron_utils.py`:

1. **Copy GPU bf16 grads → CPU fp32 main_param grads** (small transfer, ~4 GiB)
2. **CPU AdamW update** (exp_avg, exp_avg_sq, main_params all on CPU, ~16 GiB)
3. **Copy updated CPU fp32 params → GPU bf16 model params** (small transfer, ~4 GiB)

Peak GPU memory = bf16 params (~4.16 GiB) + activations (~16 GiB) = **20.04 GiB** (fits in 24 GiB!)

| Config | Layers | Optimizer | Peak GPU Memory | Status |
|--------|--------|-----------|-----------------|--------|
| v13    | 4      | GPU (FusedAdam) | 17.35 GiB | PASS |

CPU Adam reduces peak GPU by eliminating optimizer state from GPU entirely. 8-layer case even has lower peak than 4-layer GPU optimizer because CPU Adam doesn't need to load optimizer state onto GPU at any point.

## Other Changes

- **megatron_actor.py**: Dispatch to `cpu_adam_step` when `optimizer_offload=True` (config flag, not device-type check — offloaded tensors still report 'cuda' after storage resize to 0)
- **megatron_workers.py**: Simplified optimizer offload (just `offload_megatron_optimizer`, no pre-allocation); removed load/offload around `update_actor`
- **gated_delta_net.py**: Chunked cumsum (CHUNK_SIZE=8) to reduce peak activation memory during DeltaNet backward pass
- **attention.py**: Downgraded prefix hook logs from `warning` to `debug/info` (61 "prefix hook failed" warnings per step were noise, not errors — the hook correctly falls through when no prefix-sharing context is active)
- **optimizer.py**: Skip `count_zeros_in_grad` (returns 0) for speed

## PG 竞品分析 E2E (今天补充)

今天完成了 PrefixGrouper (PR #4368) 的 GRPO 端到端竞品分析。

**关键结果 (Qwen2.5-0.5B-Instruct + RTX 4090)**:

| Metric | No-PG | PG | Speedup |
|--------|-------|-----|---------|
| Training fwd+bwd | 220.8ms | 226.6ms | **0.97x** (PG slower!) |
| Peak memory | 17.22 GiB | 18.21 GiB | **-5.9%** (PG uses MORE!) |
| Full GRPO step | 10.248s | 10.254s | **1.00x** (no benefit!) |
| Rollout fraction | 97.8% | 97.8% | — |

**关键洞察**:
- **Rollout 占 97.8% 的 step 时间!** 即使 PG 将训练加速到不可能的 10x, step 也仅加速 1.20x
- PG 仅优化 attention (~8-15% 的计算量), 不优化 MLP (~68-80%)
- 在 n=4 + 0.5B 模型上, PG 的 2x kernel launch overhead 反而使训练更慢

**对 PS 的影响**:
- PS 也有同样的 rollout 瓶颈问题 → 需要 async rollout 才能体现训练加速
- 在 sync rollout (colocate) 模式下, PS 的训练加速被 97%+ rollout 时间完全稀释
- PR 作者在 8B+H800 上报告的加速, 可能使用了 async rollout (rollout 在独立 GPU)

- **PS performance benchmark (verl monkey-patch approach)**: Final benchmark with real weights + CPU AdamW:
  - Precision: cos_sim=0.998511 PASS (much better than previous NaN from random DeltaNet weights)
  - PS OFF avg time: 43.312s (steps 1+)
  - PS ON avg time: 37.287s (steps 1+) → **1.16x faster!**
  - Memory: 11.22GB both (CPU AdamW keeps optimizer state off GPU → no memory difference)
  - Previous manual approach was 0.69-0.79x slower; verl monkey-patch approach is 1.16x faster
  - Key insight: PS patches inside model.forward() are more efficient than manual Python loops

- **PS diagnostic analysis (planner + batch format)**:
  - Diagnostic revealed critical PS planner bug: only 1/4 sequences detected as reuser!
  - Planner output for P=64 N=4: `is_provider=[True,True,True,False]`, `prefix_lens=[0,0,0,64]`
  - Only Row 3 is a reuser (prefix_len=64), Rows 0-2 are standalone (prefix_len=0)
  - All 4 sequences share the same 64-token prefix, but planner only detects 1 reuser
  - Root cause: TriePrefixDetector grouping bug — doesn't group all N sequences together
  - Fix: modify `build_prefix_sharing_micro_batch` to trim ALL rows to suffix-only
  - Before fix: valid_tokens=[128,192,192,128]=640 total → 8.3% token savings
  - After fix: valid_tokens=[128,128,128,128]=512 total → 25% token savings

- **PS benchmark v4 (all-suffix trimming + fixed loss)**:
  - P=64 S=128 N=4: cos_sim=0.997421, speedup **1.04x** (near parity), mem **+2.1%**
  - P=256 S=128 N=4: speedup **0.98x** (near parity), mem **+21.2%** (17.18→13.53GB)
  - P=512 S=128 N=4: PS OFF OOM! PS ON **PASS** (cos_sim=0.995079, peak=21.72GB, 60% token savings)
  - P=256 S=128 N=8: PS OFF OOM! PS ON **PASS** (cos_sim=0.995370, peak=21.45GB, 58.3% token savings)
  - P=64 S=128 N=8: PS ON **1.09x faster** + **10.1% memory savings** (17.62→15.84GB), cos_sim=0.997375 PASS
  - All-suffix trimming confirmed: valid_tokens=[128,128,128,128] per row (N=4/N=8)
  - **Key insight**: PS ON is a **memory optimization that enables configs PS OFF can't run**, AND provides measurable speedup at larger N
  - N=4: speed near parity (0.98-1.04x); N=8: 1.09x faster + 10.1% memory savings
  - P=512/P=256: PS OFF OOMs, PS ON fits → **enables training impossible without PS**
  - Loss computation fix: slice logits to suffix positions before log_softmax (avoids OOM at N=8)
