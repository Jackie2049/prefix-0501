## TriePrefixDetector Root Cause Fix (commit 9bb6e5d)

**Problem**: Planner only detected 1/4 sequences as reusers despite all sharing the same prefix.

**Root cause**: `min_group_size=N` in TriePrefixDetector. The trie is **online/incremental** — it processes sequences one-by-one, inserting each into the trie. With min_group_size=4:
- Sequence 0: trie empty, no match → standalone (is_provider=True)
- Sequence 1: matches sequence 0 in trie (matched_group_size=2) → **2 < 4 = FAIL** → standalone!
- Sequence 2: matches (matched_group_size=3) → **3 < 4 = FAIL** → standalone!
- Sequence 3: matches (matched_group_size=4) → **4 >= 4 = PASS** → reuser

Only the last sequence passes the check. Earlier sequences can't see enough prior entries.

**Fix**: Override `min_group_size=2` in `build_prefix_sharing_micro_batch`. Any sequence that shares a prefix ≥ min_prefix_len with ANY earlier sequence becomes a reuser. The online trie naturally handles this: each sequence after the provider will see at least 1 prior entry, giving matched_group_size=2.

**Before fix**: `is_provider=[True,True,True,False]`, `prefix_lens=[0,0,0,64]`
**After fix**: `is_provider=[True,False,False,False]`, `prefix_lens=[0,64,64,64]`

## PG E2E GRPO Competitive Analysis (verl pipeline, Qwen2.5-0.5B-Instruct)

Used verl's GRPO training pipeline with HF rollout on RTX 4090. Enhanced standalone script v6 with PrefixGrouper monkey-patch for proper PG ON support.

**PG OFF Results (BATCH=4 prompts, N=4, 5 steps)**:

| Phase | Avg Time | % of Step |
|-------|----------|-----------|
| Rollout | 29.65s | **99.3%** |
| LogProb | 0.072s | 0.2% |
| Advantages | 0.016s | 0.1% |
| Train (fwd+bwd+opt) | 0.12s | 0.4% |
| **Step total** | **29.86s** | — |

**Peak GPU memory**: 12.99 GB (model + ref_model + activations)

**PG ON Results (BATCH=4 prompts, N=4, 5 steps, PrefixGrouper monkey-patch + flash_attention_2)**:

| Phase | Avg Time | % of Step | vs PG OFF |
|-------|----------|-----------|-----------|
| Rollout | 38.41s | **99.4%** | 1.29x slower! |
| LogProb (PG) | 0.123s | 0.3% | 1.71x slower |
| Train | 0.088s | 0.2% | 0.71x faster |
| **Step total** | **38.64s** | — | **1.29x slower!** |

**Peak GPU memory**: 12.99 GB

**Key findings**:
1. **PG ON is 1.29x slower overall!** — rollout dominates >99%, PG makes logprob 1.71x slower (grouped attention overhead)
2. **PG training portion is 0.71x faster** (1.41x speedup) — but this only saves 0.035s out of 29.86s
3. **Same rollout bottleneck confirmed**: both PG OFF (99.3%) and PG ON (99.4%) show rollout dominates
4. **PG ON total step 38.64s vs PG OFF 29.86s** — PG actually HURTS performance by making logprob slower
5. **Even if PG makes training infinitely fast**: step speedup = 1/(0.993 + 0.007/∞) ≈ **1.007x** — negligible

**PG ON E2E with verl's actual Ray pipeline failed** due to:
- tensordict 0.7.2 serialization incompatibility (DataProto.batch becomes None after Ray serialization)
- RolloutConfig rejects sync mode (patched but not fully working)
- RolloutReplicaRegistry doesn't support HF (would need full async rollout infrastructure)
- HFRollout abstract method issues with new BaseRollout base class

**verl Ray pipeline patches applied** (not fully working):
- base.py: Added HFRolloutAdapter to _ROLLOUT_REGISTRY for ("hf", "sync") mode
- rollout.py: Allow sync mode only for HF rollout (name="hf")
- ray_trainer.py: Set async_rollout_mode based on rollout.name (False for HF)
- fsdp_workers.py: Skip rollout_mode() weight transfer for HF rollout
- hf_rollout.py: Remove BaseRollout inheritance (avoid abstract method issues)

**Implication for PS**: PS has the same rollout bottleneck → **async rollout is required** for any meaningful step-level speedup. PS's 1.12x training speedup at N=4 translates to only ~1.004x step speedup in colocate mode.
