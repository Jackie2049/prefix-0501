## TriePrefixDetector Root Cause Fix (commit 9bb6e5d)

**Problem**: Planner only detected 1/4 sequences as reusers despite all sharing the same prefix.

**Root cause**: `min_group_size=N` in TriePrefixDetector. The trie is **online/incremental** — it processes sequences one-by-one, inserting each into the trie. With min_group_size=4:
- Sequence 0: trie empty, no match → standalone (is_provider=True)
- Sequence 1: matches sequence 0 in trie (matched_group_size=2) → **2 < 4 = FAIL** → standalone!
- Sequence 2: matches (matched_group_size=3) → **3 < 4 = FAIL** → standalone!
- Sequence 3: matches (matched_group_size=4) → **4 >= 4 = PASS** → reuser

Only the last sequence passes the check. Earlier sequences can't see enough prior entries.

**Fix**: Override `min_group_size=2` in `build_prefix_sharing_micro_batch`. Any sequence that shares a prefix ≥ min_prefix_len with ANY earlier sequence becomes a reuser. The online trie naturally handles this: each sequence after the provider will see at least 1 prior entry, giving matched_group_size=2.

**Before fix**: `is_provider=[True,True,True,False]`, `prefix_lens=[0,0,0,64]`
**After fix**: `is_provider=[True,False,False,False]`, `prefix_lens=[0,64,64,64]`

### Why Full verl Pipeline Cannot Run (Infrastructure Limitation)

**vLLM is hard-locked to torch==2.6.0**: vLLM 0.8.5's `requirements/cuda.txt` specifies `torch==2.6.0` as a hard requirement. Our server has torch 2.9.0+cu128 (required by our prefix-0501 project). Downgrading torch would break our entire development environment.

**Additional blockers confirmed**:
1. vLLM ABI incompatible: pre-built vllm 0.8.5+cu121 wheel has undefined symbol `parseSchemaOrName` with torch 2.9.0 (different C++ ABI)
2. Building vLLM from source requires many dependencies (sentencepiece, blake3, etc.) that can't be downloaded on server (campus auth blocks pip)
3. SGLang 0.5.6 (alternative async rollout) doesn't exist on PyPI (only up to 0.4.10.post2 + 0.5.0rc0)
4. HF rollout not registered in `_ROLLOUT_REGISTRY` (only `("vllm","async")` and `("sglang","async")`)
5. RolloutConfig rejects `mode="sync"` — only `mode="async"` supported in PR #4368

**Impact**: This version lock-in is a **real-world deployment limitation** for PR #4368. Users with newer torch versions (2.7+, 2.9+) cannot use PG's actual verl pipeline. This is a competitive disadvantage vs PS (which works with any torch/attn version).

**v8 standalone benchmark** is the best E2E test possible. It uses verl's same PG monkey-patch code (`apply_prefix_grouper_patch`, `build_pg_from_micro_batch`, `pg_forward`) and faithfully reproduces GRPO training (rollout → logprob → advantage → loss+backward).

**Conclusion**: Cannot run full verl Ray-based pipeline. v8 standalone benchmark uses same PG monkey-patch + prefix_grouper_utils (build_pg_from_micro_batch, pg_forward) — faithful reproduction of PG's core optimization.

PG's benefit is only in the logprob + train portions (NOT rollout):

| Config | PG OFF LogProb | PG ON LogProb | PG OFF Train | PG ON Train | LogProb Speedup | Train Speedup |
|--------|---------------|---------------|-------------|-------------|-----------------|---------------|
| N=4 P=64 | 0.090s | 0.139s | 0.136s | 0.099s | 0.65x (slower!) | 1.37x |
| N=4 P=128 | 0.097s | 0.139s | 0.168s | 0.101s | 0.70x (slower!) | 1.66x |
| N=4 P=256 | 0.121s | 0.140s | 0.249s | 0.102s | 0.86x | **2.45x** |
| N=8 P=64 | 0.094s | 0.147s | 0.136s | 0.096s | 0.64x (slower!) | 1.42x |

**Critical finding**: PG makes **logprob slower** (0.64-0.86x) because grouped attention overhead exceeds prefix sharing savings for short prompts. PG only helps the train (backward) portion because prefix sharing reduces gradient computation for shared prefix tokens.

**Why PG logprob is slower**: PG decomposes into prefix self-attn + suffix concat-attn. This requires:
1. Two separate forward passes (prefix + suffix) vs one pass (normal)
2. Concat-attention has larger K/V matrices (suffix + prefix) per sequence
3. Overhead exceeds savings when prefix is short (P=64, 64 tokens)

Regardless of PG/PS, rollout dominates >99% of step time in colocate mode:

| Mode | Rollout% of Step | PG Step Impact |
|------|-----------------|----------------|
| PG OFF (flash) | 99.1-99.5% | Baseline |

**Maximum theoretical step speedup**: If PG makes logprob+train infinitely fast:
- speedup = 1 / (rollout% + 0%) ≈ 1/0.995 ≈ **1.005x**
- This is negligible regardless of PG's training speedup.

**The ONLY path to meaningful step-level speedup is async rollout** (separate GPU for generation), which removes rollout from step time measurement. This applies equally to PG and PS.
