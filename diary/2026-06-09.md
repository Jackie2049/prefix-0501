## TriePrefixDetector Root Cause Fix (commit 9bb6e5d)

**Problem**: Planner only detected 1/4 sequences as reusers despite all sharing the same prefix.

**Root cause**: `min_group_size=N` in TriePrefixDetector. The trie is **online/incremental** — it processes sequences one-by-one, inserting each into the trie. With min_group_size=4:
- Sequence 0: trie empty, no match → standalone (is_provider=True)
- Sequence 1: matches sequence 0 in trie (matched_group_size=2) → **2 < 4 = FAIL** → standalone!
- Sequence 2: matches (matched_group_size=3) → **3 < 4 = FAIL** → standalone!
- Sequence 3: matches (matched_group_size=4) → **4 >= 4 = PASS** → reuser

Only the last sequence passes the check. Earlier sequences can't see enough prior entries.

**Fix**: Override `min_group_size=2` in `build_prefix_sharing_micro_batch`. Any sequence that shares a prefix ≥ min_prefix_len with ANY earlier sequence becomes a reuser. The online trie naturally handles this: each sequence after the provider will see at least 1 prior entry, giving matched_group_size=2.

**Before fix**: `is_provider=[True,True,True,False]`, `prefix_lens=[0,0,0,64]`
**After fix**: `is_provider=[True,False,False,False]`, `prefix_lens=[0,64,64,64]`

5. **Even if PG makes training infinitely fast**: step speedup = 1/(0.993 + 0.007/∞) ≈ **1.007x** — negligible

## PG v7 Multi-Config Sweep (Qwen2.5-0.5B, RTX 4090, colocate HF rollout)

Extended v6 standalone script to sweep N=2,4,8 and P_LEN=64,128.

**v7 Results Summary**:

| Config | PG OFF Step | PG ON Step | Step Speedup | Train Speedup | Rollout% (OFF) | Rollout% (ON) | OFF Mem | ON Mem |
|--------|-------------|------------|--------------|---------------|----------------|---------------|---------|--------|
| N=2 P=64 | 15.16s | 19.71s | **0.77x (slower!)** | 0.69x | 99.2% | 99.0% | 9.13GB | 9.80GB |
| N=4 P=64 | 30.56s | 38.81s | **0.79x (slower!)** | 1.40x | 99.3% | 99.4% | 12.98GB | 13.00GB |
| N=8 P=64 | 28.01s | 37.22s | **0.75x (slower!)** | 1.42x | 99.2% | 99.4% | 12.98GB | 11.57GB |
| N=4 P=128 | 29.01s | 41.36s | **0.70x (slower!)** | 1.80x | 99.1% | 99.4% | 15.27GB | 13.36GB |

**Key findings**:
1. **PG ON is consistently SLOWER overall**: 0.70-0.79x across all configs
2. **Training portion faster**: 1.4-1.8x PG train speedup, but saves <0.5s out of 15-40s total
3. **Rollout ALWAYS dominates >99%**: regardless of N or P_LEN, rollout is the bottleneck
4. **PG ON rollout is slower**: PG ON requires flash_attention_2 backend, which makes rollout slower than SDPA
5. **Longer prompts amplify PG slowdown**: P=128 gives 0.70x (worst) vs P=64 gives 0.77-0.79x
6. **PG ON saves memory at N=8**: 11.57GB vs 12.98GB (PG groups shared prefix → fewer activations)

**Comparison with PR #4368's claimed results**:
- PR claims 1.14x step speedup at 4K context, 1.27x at 8K (Qwen3-4B, 4×H800, vLLM async rollout)
- Our results: 0.70-0.79x step slowdown (Qwen2.5-0.5B, 1×RTX 4090, HF rollout, colocate mode)
- **Why the difference**: PR uses vLLM async rollout (separate GPU for generation), which removes rollout from step time measurement. In colocate mode, rollout can't be hidden.

**Implication for PS**: PS has the same rollout bottleneck → **async rollout is required** for any meaningful step-level speedup. PS's 1.12x training speedup at N=4 translates to only ~1.004x step speedup in colocate mode.

**attn_impl Comparison (PG OFF only)**:
- sdpa: 32.62s step, 32.40s rollout
- flash_attn: 43.77s step, 43.53s rollout
- **sdpa is 1.34x faster than flash_attn** for Qwen2.5-0.5B rollout!

**v8 Key Findings**:
1. **Fair comparison: PG ON at near parity (0.95-1.05x)** — v7 showed 0.70-0.79x because sdpa is 1.34x faster than flash_attn!
2. **PG ON requires flash_attn** → loses sdpa speed advantage → in production with sdpa baseline, PG ON is net slower
3. **PG ON saves significant memory at longer prompts**: P=256 → 27.7% memory savings (19.7→14.2GB)
4. **Rollout still dominates 99%** → PG's train/logprob improvements are negligible in step time
5. **PG ON rollout is not accelerated** (it uses the same generation path) → PG only helps training portion

**Comparison: PG vs PS**:
- **PG**: near parity step speedup, 27% memory savings at P=256, requires flash_attn, requires model modification
- **PS**: 1.12x training speedup, enables OOM configs, works with any attn_impl, no model modification needed
- **Both**: rollout dominates → async rollout is the only path to meaningful step-level speedup
