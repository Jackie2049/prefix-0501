## TriePrefixDetector Root Cause Fix (commit 9bb6e5d)

**Problem**: Planner only detected 1/4 sequences as reusers despite all sharing the same prefix.

**Root cause**: `min_group_size=N` in TriePrefixDetector. The trie is **online/incremental** — it processes sequences one-by-one, inserting each into the trie. With min_group_size=4:
- Sequence 0: trie empty, no match → standalone (is_provider=True)
- Sequence 1: matches sequence 0 in trie (matched_group_size=2) → **2 < 4 = FAIL** → standalone!
- Sequence 2: matches (matched_group_size=3) → **3 < 4 = FAIL** → standalone!
- Sequence 3: matches (matched_group_size=4) → **4 >= 4 = PASS** → reuser

Only the last sequence passes the check. Earlier sequences can't see enough prior entries.

**Fix**: Override `min_group_size=2` in `build_prefix_sharing_micro_batch`. Any sequence that shares a prefix ≥ min_prefix_len with ANY earlier sequence becomes a reuser. The online trie naturally handles this: each sequence after the provider will see at least 1 prior entry, giving matched_group_size=2.

**Before fix**: `is_provider=[True,True,True,False]`, `prefix_lens=[0,0,0,64]`
**After fix**: `is_provider=[True,False,False,False]`, `prefix_lens=[0,64,64,64]`

## PG E2E GRPO Competitive Analysis (verl pipeline, Qwen2.5-0.5B-Instruct)

Used verl's GRPO training pipeline with HF rollout on RTX 4090. Standalone script that runs full GRPO loop: rollout → logprob → advantages → training update.

**PG OFF Results (BATCH=4 prompts, N=4, 3 steps)**:

| Phase | Avg Time | % of Step |
|-------|----------|-----------|
| Rollout | 29.75s | **99.3%** |
| LogProb | 0.073s | 0.2% |
| Advantages | 0.016s | 0.1% |
| Train (fwd+bwd+opt) | 0.12s | 0.4% |
| **Step total** | **29.96s** | — |

**Peak GPU memory**: 13.14 GB (model + ref_model + activations)

**PG ON was not completed** — HuggingFace's standard Qwen2 model doesn't support PrefixGrouper's `prefix_grouper` parameter in the model forward call. PG requires verl's custom model wrapper (`use_remove_padding=False` + `prefix_grouper` kwarg). The standalone HuggingFace model only supports standard attention, not PG's grouped prefix-suffix decomposition.

**Key finding (same as 2026-06-08)**:
- **Rollout occupies >99% of step time!** Even if PG/PS makes training infinitely fast, step speedup = 1/(0.993 + 0.007/∞) ≈ **1.007x** — negligible
- This confirms the rollout bottleneck: in colocate (sync) mode, rollout generation dominates
- PG's value proposition is the same as PS: **only matters with async rollout** (separate GPU for generation)
- The training portion (0.19s out of 29.96s) is so tiny that even 10x training speedup → only 1.06x step speedup

**Comparison with previous PG analysis** (2026-06-08 standalone benchmark):
- Previous: rollout 97.8% (vllm rollout with n=5, 0.5B model, 8 GPUs)
- This run: rollout 99.3% (HF rollout with n=4, 0.5B model, 1 GPU)
- Consistent: rollout always dominates in colocate mode regardless of rollout backend

**Implication for PS**: PS has the same rollout bottleneck → **async rollout is required** for any meaningful step-level speedup. PS's 1.12x training speedup at N=4 translates to only ~1.004x step speedup in colocate mode.
