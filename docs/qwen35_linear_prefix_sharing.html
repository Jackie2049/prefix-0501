Linear Attention：递推式

每个新 token 把自己的 K、V "融合"进一个固定大小的状态矩阵 S。不看历史表，只看当前摘要。

新 token 的 K, V
K:[32,128] V:[32,128]
→ 融入 →
状态矩阵 S
[32, 128, 128]
固定大小！永远不变！
→
新 token 的 Q
[32, 128]
→ Q×S →
输出
[32, 128]

递推公式——状态 S 是怎么一步步更新的

处理第 t 个 token 时，有三个"门"控制状态更新：

St = Gt ⊙ St-1 + βt × KtT ⊗ (Vt - St-1 @ Kt)
Gt = 衰减门
exp(-exp(A) × dt)
越久远的信息衰减越多
βt = 写入门
sigmoid(b)
控制当前 token 写入多少新信息
(V - S@K) = 修正项
Delta Rule
只写入"预测误差"，而非原始 V

展开递推看，处理完 N 个 prefix token 后的状态 SN：

SN = Σt=1..N (Πj=t+1..N Gj) × βt × KtT × ΔVt

SN 是 N 个 prefix token 信息的加权和，压缩在一个 [B, 32, 128, 128] 的固定矩阵里。

★
关键性质：SN 是 prefix 的"充分统计量"。
从 SN 开始继续处理 suffix token，数学上等价于从 S0=0 开始处理完整序列 [prefix + suffix]。
这就是 prefix sharing 可以工作的数学基础。

02Linear Attention 层的前缀共享流程

分 Phase 0 和 Phase 1。核心：把 prefix 处理完的状态 Sprefix 拷贝给每条 reuser 序列。

Phase 0 — 处理 prefix tokens，得到 Sprefix 和 Cprefix

只取 1 份公共 prefix tokens，送入每层的 GatedDeltaNet，保存处理后的状态。

  1. 取出 prefix tokens，做 QKV + z + b + a 投影

    和正常 forward 的投影完全一样，只是输入只有 prefix 部分。

    输入
    [1, P, 4096]
    →
    in_proj_qkv
    [1, P, 8192]
    in_proj_z
    [1, P, 4096]
    in_proj_b
    [1, P, 32]
    in_proj_a
    [1, P, 32]

  2. Causal Conv1D 预处理（带状态）

    1D 因果卷积，kernel=4。处理完后保存卷积状态（最后 3 个 token 的特征）。

    conv 后的 Q
    [1, P, 16, 128]
    conv 后的 K
    [1, P, 16, 128]
    conv 后的 V
    [1, P, 32, 128]
    Cprefix（要保存！）
    [1, 8192, 3]
    ← 固定大小，供 Phase 1 的 Conv1D 初始化

  3. GatedDeltaNet 递推：从 S0=0 开始，处理 P 个 token

    调用 chunk_gated_delta_rule(initial_state=零矩阵, output_final_state=True)

    S0（初始）
    全零 [1, 32, 128, 128]
    → 处理 P 个 token →
    Sprefix（要保存！）
    [1, 32, 128, 128]

  4. 保存两个状态到状态池

    写入预分配的 ssm_state 和 conv_state 张量池，按 slot 索引存储。

    ssm_state[provider_slot] = Sprefix   # [1, 32, 128, 128]
    conv_state[provider_slot] = Cprefix   # [1, 8192, 3]

Phase 1 — 拷贝状态，每条序列只算 suffix

  1. 把 Sprefix 和 Cprefix 拷贝到每个 reuser 的 slot

    纯 GPU 显存拷贝，极快（每个 slot ~1MB）。

    # 拷贝给所有 reuser
    ssm_state[reuser_slot] = ssm_state[provider_slot]
    conv_state[reuser_slot] = conv_state[provider_slot]

  2. 标记 reuser "有初始状态"

    has_initial_state[reuser_idx] = True

    设为 True 后，GDN forward 就不会清零 initial_state，而是直接用里面已有的 Sprefix。

  3. 正常 forward suffix tokens

    GDN forward 自动执行以下逻辑（vLLM 已实现，我们不用改）：

    # 从状态池取出当前序列的状态
    initial_state = ssm_state[slot_idx] # ← 就是 Sprefix [1, 32, 128, 128]

    # has_initial_state=True → 不清零
    initial_state[~has_initial_state] = 0 # reuser 被跳过

    # 跑 GDN，从 Sprefix 开始处理 suffix
    output, S_final = chunk_gated_delta_rule(
        q, k, v, g, beta, # suffix tokens 的投影结果
        initial_state=initial_state, # ← 非零！[1, 32, 128, 128]
        output_final_state=True,
    )
    # output: [1, S_suffix, 32, 128]
    # S_final: [1, 32, 128, 128] (prefix+suffix 合并后的完整状态)

    # 写回最终状态
    ssm_state[slot_idx] = S_final
✓
Phase 1 中，Provider 和 Reuser 的计算路径完全相同。
区别仅在于：Provider 的 slot 里有 Phase 0 写入的 Sprefix，Reuser 的 slot 里是我们拷贝过去的同一份 Sprefix。
GDN forward 不知道也不关心这个区别，它只看 has_initial_state 标记和 slot 里的值。

为什么 Linear Attention 路径直接 return None？
因为 vLLM 的 GDN forward 已经实现了 has_initial_state 机制。 我们在 Phase 0 把 Sprefix 写入了 ssm_state 池， 在进入 forward 前把 Sprefix 拷贝到了 reuser 的 slot 并设置了 has_initial_state=True。 GDN forward 自己会读取 slot 里的状态、不清零、从 Sprefix 开始递推。 所以我们不需要拦截 GDN 的 forward，让它自己跑就行。
