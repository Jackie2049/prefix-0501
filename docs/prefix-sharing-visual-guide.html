1. 问题：为什么需要 Prefix Sharing

RL 训练中同 Batch 内多条轨迹共享 Prompt 前缀
P(Provider) 正常计算；R(Reuser) 复用 P 的前缀 KV；S(Standalone) 无共享，正常计算

问题 在 agentic RL 的 tree/step 模式下，同 batch 内 60-80% 的 token 是共享前缀，前缀 KV 重复计算浪费大量算力。

方案 Provider 只计算一次前缀 KV，Reuser 注入（inject）该 KV，只计算自己的后缀。

FlashAttentionBackend 关键逻辑伪代码
总结：Prefix Sharing 的核心是在 Attention 计算前拦截，Provider 算一次前缀 KV 存入 Store，Reuser 拼接 Provider 的前缀 KV + 自己的后缀 KV，用自定义 Causal Mask 做 Attention。梯度通过 torch.cat（不 detach）自然回传。适配 Flash Attention 的最简方案是 Q Padding：Reuser 的 Q 前面填零使其与 KV 等长，复用 FA 的 causal=True，最后裁剪输出。
