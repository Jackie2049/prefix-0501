Prefix Sharing 梯度回传正确性分析 严格数学证明 + 代码路径追踪

0. 分析设定

考虑一个 micro-batch 内两条序列，Provider P 和 Reuser R，共享前缀长度 prefix_len = p，各自总长分别为 L_P 和 L_R。

关键前提：P 和 R 的前缀 token 完全相同（由 TriePrefixDetector 精确保证），经过同一模型、同一层、同一参数，所以在每一层 l：

h_P^{(l)}[0..p) = h_R^{(l)}[0..p) ← 隐藏状态完全相同（归纳法可证） K_P^{(l)}[0..p) = K_R^{(l)}[0..p) ← 因此 KV 也完全相同 V_P^{(l)}[0..p) = V_R^{(l)}[0..p)

这个等价性是整个梯度正确性的基石。下面逐环节分析。

1. PrefixKVStore：梯度是否保留？

代码追踪：store 存的是什么？

结论：梯度保留 ✓

Store 只保存 Python 引用，没有任何 detach()、clone() 或序列化操作。Provider 的 K/V tensor 在存入和取出时 autograd 计算图完全一致。

2. build_kv()：torch.cat 的梯度行为

KV 扩展的两种路径
数学验证：

令 expanded_K = torch.cat([K_P[:p], K_R_s], dim=0)，其中 K_P 是 Provider 的 key，K_R_s 是 Reuser 的 suffix key。

Backward 时，PyTorch autograd 对 torch.cat 的梯度处理是：将输出梯度沿 cat 的分界点拆分回各输入。

因此：∂L/∂K_P[:p]（来自 R 的 attention）和 ∂L/∂K_P（来自 P 自己的 attention）会自动累加到同一个 tensor 上。

结论：torch.cat 梯度正确 ✓

PyTorch 的 torch.cat 原生支持 autograd。梯度会沿 cat 边界正确拆分回各输入 tensor。Provider 的 KV 同时接收来自自身 attention 和所有 Reuser attention 的梯度，自动累加。

3. Attention 计算：逐行显式计算是否有梯度问题？

当前实现：显式 einsum + softmax

关键点 masked_fill 用 -inf（即 finfo.min）填充 masked 位置。Softmax 对 -inf 输出 0，backward 时这些位置的梯度也为 0，不会影响可见区域的梯度。

结论：Attention 计算梯度正确 ✓

全部使用标准可微操作（einsum、masked_fill、softmax），PyTorch autograd 原生支持。custom causal mask 不引入任何不可导操作。

4. 核心证明：梯度是否与原始（无 prefix sharing）等价？

这是最关键的问题。令 θ 为模型参数（如 W_k），证明：

∂(L_P + L_R) / ∂θ 在 prefix-sharing 下 == 原始（无共享）下

4.1 原始情况（无 prefix sharing）

K_P = W_k · h_P ← Provider 的 KV，经过 P 的完整 forward K_R = W_k · h_R ← Reuser 的 KV，经过 R 的完整 forward（含前缀） out_P = Attn(Q_P, K_P, V_P, causal_mask) out_R = Attn(Q_R, K_R, V_R, causal_mask) L = L_P(out_P) + L_R(out_R) ∂L/∂W_k = ∂L_P/∂K_P · ∂(W_k·h_P)/∂W_k + ∂L_R/∂K_R · ∂(W_k·h_R)/∂W_k = ∂L_P/∂K_P · h_P^T + ∂L_R/∂K_R · h_R^T 其中 ∂L_R/∂K_R 包含前缀和后缀两部分位置

4.2 Prefix Sharing 情况

K_P = W_k · h_P ← Provider 不变 K_R_s = W_k · h_R_s ← Reuser 只算 suffix K_R_exp = cat([K_P[:p], K_R_s]) ← 注入 Provider 前缀 out_P = Attn(Q_P, K_P, V_P, causal) ← Provider 不变 out_R = Attn(Q_R_s, K_R_exp, V_R_exp, shifted_causal) L = L_P(out_P) + L_R(out_R)
证明链：

Step 1：前缀 token 完全相同 → Layer 0 的 embedding 相同 → Layer 0 的 hidden state 相同

Step 2（归纳假设）：若 Layer l-1 的 h_P[:p] = h_R[:p]，则 Layer l 的 KV 有：

K_P^{(l)}[:p] = W_k · h_P^{(l)}[:p] = W_k · h_R^{(l)}[:p] = K_R^{(l)}[:p] 因为同一模型、同一参数、同一输入 → 同一输出

Step 3：R 的 attention 在 prefix-sharing 下使用的 K_R_exp = cat([K_P[:p], K_R_s])，而原始的 K_R = [K_R[:p], K_R[p:]]。由于 K_P[:p] = K_R[:p]，所以 K_R_exp == K_R。

Step 4：因此 R 的 attention 输出完全相同 → R 的 loss 完全相同 → ∂L_R/∂K_R_exp 在数值上等于 ∂L_R/∂K_R。

Step 5：梯度通过 torch.cat 拆分：

∂L/∂W_k = ∂L_P/∂K_P · h_P^T + ∂L_R/∂K_R_exp · h_R_exp^T = ∂L_P/∂K_P · h_P^T + ∂L_R/∂K_P[:p] · h_P[:p]^T + ∂L_R/∂K_R_s · h_R_s^T = ∂L_P/∂K_P · h_P^T + ∂L_R/∂K_R[:p] · h_R[:p]^T + ∂L_R/∂K_R[p:] · h_R[p:]^T 因为 K_P[:p]=K_R[:p], h_P[:p]=h_R[:p], ∂L_R/∂K_P[:p]=∂L_R/∂K_R[:p] = ∂L_P/∂K_P · h_P^T + ∂L_R/∂K_R · h_R^T = 原始梯度 ✓

结论：梯度数学等价 ✓

因为 P 和 R 的前缀 token 完全相同（由 TriePrefixDetector 精确保证），经过同一模型同一参数，每一层的隐藏状态、KV 都完全相同。因此 torch.cat([K_P[:p], K_R_s]) 等价于原始的 K_R，梯度在数值上完全一致。

5. 多个 Reuser 共享同一 Provider 的梯度累加

场景：1 个 Provider P，N 个 Reuser R1...RN 共享同一前缀

结论：多 Reuser 梯度累加正确 ✓

PyTorch autograd 在同一个 tensor 上自动累加来自多个消费者的梯度。K_P 被 P 自身 + N 个 Reuser 的 attention 共同引用，backward 时梯度正确累加。

6. Prefix-Last Logprob 恢复的梯度

恢复 Reuser 第一个 suffix token 的 logprob
为什么要 .clone() 而不是 .detach()？

.clone() 创建 tensor 副本但保留 autograd 计算图。梯度从 restored logprob 回传时会经过 provider_logits 的计算图，最终到达模型参数。

如果用 .detach()，梯度会被截断，Provider 的前缀参数就收不到来自 Reuser 首 token 的梯度信号。

结论：Logprob 恢复梯度正确 ✓

.clone() 保留计算图，梯度从 Reuser 的 loss 流回 Provider 的 logits，再到模型参数。链路完整。

7. 潜在风险与边界情况

7.1 Store 生命周期：梯度会丢失吗？

context.py 中的生命周期管理

安全 store.close() 在 finally 块中执行，即 context manager 退出时才清空。只要 with 块包含了整个 forward + backward 过程，梯度就不会丢失。

注意 如果 backward() 在 with 块之外调用，Store 已经被 close()，但 close() 只是清空了字典引用，不会删除 tensor 本身。autograd 图中仍然持有 Provider K/V 的引用，梯度仍能正确回传。但 _ensure_open() 会阻止新的 store/load 操作。

7.2 RoPE 动态扩展的梯度

RoPE 扩展：extra_angles = extra_positions[:, None, None, None] * step

step = q_pos_emb[1] - q_pos_emb[0]（两个位置的差值 = inv_freq），extra_positions 是整数 arange（不需要梯度）。

梯度只通过 step 流回 q_pos_emb。由于 RoPE 的数学性质 freqs[p] = p × inv_freq（线性），这个外推是精确的，不会引入近似误差。

7.3 混合精度（fp16/bf16）下的数值差异

潜在风险 当前实现使用 torch.einsum + softmax 的显式计算（O(N²) 显存），而原始 Megatron 可能使用 Flash Attention（fused kernel）。两者的 softmax 实现可能有微小的数值差异（约 1e-3 级别），在 fp16 下可能累积。

但这不影响梯度的数学正确性，只影响数值精度。对训练稳定性的影响需要在实际场景中评估。

7.4 梯度缩放 / Loss 归一化

潜在风险 Prefix Sharing 裁剪了 Reuser 的前缀 token，导致 batch 内总 token 数减少。如果 loss 按总 token 数归一化：

  • 原始：loss = sum(per_token_loss) / total_original_tokens
  • 裁剪后：实际计算的 token 数 < 原始 token 数

需要确保 loss 归一化使用的是裁剪后的实际 token 数还是原始 token 数。如果框架自动按 attention_mask 中的有效 token 数归一化，则裁剪后的 loss 会被更少的 token 除，导致梯度略大。

建议：在 loss 计算前恢复原始的 token 计数，或在 loss mask 中正确标记。

7.5 内存压力

潜在风险 Provider 的 K/V tensor 需要保留到所有 Reuser 完成 backward。如果一个 Provider 被 N 个 Reuser 共享：

  • Provider 的 KV 计算图在内存中持续存在（不会被提前释放）
  • 每个 Reuser 的 torch.cat 创建一个 expanded KV，也需要内存
  • 显式 einsum attention 需要 O(N²) 的 attention matrix 显存

这是 性能问题，不是 正确性问题。适配 Flash Attention 后 O(N²) 会降为 O(N)。

最终结论：梯度回传数学正确，无本质问题

Prefix Sharing 的梯度回传在数学上是正确的，与"不使用 prefix sharing 的独立计算"完全等价。核心原因是共享前缀 token 在同一模型参数下产生完全相同的隐藏状态和 KV，因此用 Provider 的 KV 替代 Reuser 的 KV 不改变 forward 输出，也不改变 backward 梯度值。

需要关注的是三个工程层面的问题：Loss 归一化分母是否需要修正、混合精度下的数值稳定性、以及 O(N²) 的内存压力。
